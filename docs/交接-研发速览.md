# 落地页生成系统 — 研发速览（人读版）

> 5 分钟看懂这是什么、怎么跑起来、部署到你们环境要改什么。要一步步照做的完整命令，看 [交接-AGENT执行手册](交接-AGENT执行手册.md)。

## 这是什么

一个 **B2B 落地页素材生产工作台**：输入产品（公司名/品类/描述/产品图）→ 调大模型自动产出落地页素材（Banner + 英文文案 + 每个产品的产品图）→ 以两份 JSON + 图片 URL 交付给渲染端发布。有网页工作台（人工建/调任务）和一套 `/api/ext` 接口（外部平台批量建任务/查状态/拉交付）。

## 技术栈一句话

Node 22 + Express + React(Vite)，**tsx 直跑 TS（无编译）**，**文件系统存储（无数据库）**，全 Docker 化：`docker compose up` 起**一个容器**（内含 app + 内置 Xray 代理）。

## 怎么跑起来（本地/服务器都一样）

```bash
cp .env.example .env      # 填密钥（我方安全渠道单独给你真实值）
docker compose up -d --build
# 打开 http://<host>:4100 ，用 APP_PASSWORD 登录
```

## 三个必须知道的概念

1. **两个大模型来源**
   - **文本**（品类判断 + 文案）走公司 **AIGW** 网关 → 需部署机在**公司内网**能直连 AIGW。
   - **图片**默认走 **Nano Banana（Google Gemini 原生）**，**失败自动回退 AIGW 出图**。公司内网连不上 Google，所以容器**内置了 Xray 代理**专门给图片调用翻墙；AIGW/文本不走代理。
2. **一个 `.env` 管全部**：登录密码、AIGW key、Nano Banana key、Xray 节点、对接 key、域名，全在这一个文件里。改它就换配置，不用动代码。
3. **数据在 Docker 命名卷 `es-data`**（挂到容器 `/app/data`）：任务和图片都在这。`docker compose down` 不会删它，`down -v` 才会（别误用）。

## 部署到你们环境要改什么

只改 `.env` 里这几项：
- `PUBLIC_BASE_URL` = 你们的域名（产物图片对外 URL 前缀）
- `AIGW_*` = 你们内网的 AIGW 直连配置（key）
- `NANO_BANANA_API_KEY` + `XRAY_*` = 图片模型 key + 翻墙节点
- `APP_PASSWORD` / `SESSION_SECRET` / `EXT_API_KEYS` = 自定
- 反向代理/域名解析到容器的 4100 端口（你们的 nginx/网关自理）

## 历史数据

我方会给一个 `es-data.tgz`（现网**全量真实数据**，不含测试数据）。你们部署好、配好域名后，一条命令灌进去 + 自动把图片 URL 改写到你们的新域名即可全量可见（见 AGENT 手册“数据迁移”）。

## 关键红线

- 密钥只放 `.env`（已 gitignore），**绝不进镜像/仓库**。
- 交付接口是**幂等**的：外部平台入库产品必须**按 code 覆盖**，不能只插不删（详见对接文档 §5.1）。
